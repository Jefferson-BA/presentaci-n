Diagnóstico

La Problemática

Foco de dolor: por qué las arquitecturas tradicionales fallan en fases tempranas de MYPES y startups.

01

Sobreingeniería Prematura

Microservicios, Kubernetes y capas innecesarias antes de validar el producto. Complejidad que retrasa el Time-to-Market y consume presupuesto sin retorno.

02

Costos Fijos Elevados

AWS, Azure y GCP exigen compromisos mensuales de $150–$250 USD incluso con tráfico mínimo. Instancias EC2, RDS y ALB ociosas desde el día uno.

03

Bases Vectoriales Aisladas

Pinecone, Weaviate o Qdrant como servicios separados duplican costos y complejidad operativa para un chatbot que podría vivir en la misma base de datos.

04

Infraestructura Ociosa

Servidores, balanceadores y bases siempre activas consumiendo recursos en fases donde el producto aún no genera ingresos. Desperdicio financiero crítico.

03 / 13

Solución

El Camino a la Solución

Web desacoplada con Astro + React: rendimiento máximo, costo de cómputo en cliente eliminado y seguridad por diseño.

Astro + React: Separación Estática Total

El frontend se compila a HTML estático en build time. Cero servidor Node en producción para la UI. Vercel/Pages entrega assets desde CDN global con latencia mínima.

  • Islands Architecture: React solo donde se necesita interactividad
  • Build estático = imposible atacar la base de datos desde el frontend
  • Entrega inmediata: deploy en segundos tras cada push
🛡

Inmunidad a Ataques Directos

El frontend no expone credenciales ni conexiones DB. Toda lógica sensible vive en NestJS detrás de Cloudflare WAF.

⚡

Rendimiento Optimizado

Catálogo de buffets, landing y contenido estático se sirven sin JavaScript bloqueante. Core Web Vitals excelentes.

💰

Costo de Cómputo $0

Sin contenedores ni VMs para la interfaz. Solo el backend en Railway consume recursos de servidor.

🔗

Componentes Clave

Auth JWT/Google · Órdenes y reservas · Pagos Culqi · Chatbot IA con SSE · Storage R2 para imágenes.

Flujos

Casos de Uso Prácticos y Optimizados

Ejemplos reales de interacción en la plataforma De Parra Spitz.

Pago Seguro Culqi SDK/API

Procesamiento de Pago de Pedido

  1. 1 Cliente selecciona buffet y completa formulario de evento (fecha, invitados, platos).
  2. 2 Frontend Astro envía solicitud a POST /api/orders en NestJS (Railway).
  3. 3 Backend crea orden en PostgreSQL (Neon) con estado Pendiente.
  4. 4 Culqi SDK tokeniza tarjeta en el cliente — datos sensibles nunca tocan nuestro servidor.
  5. 5 NestJS confirma cargo vía API Culqi, actualiza estado a Aprobado y notifica al admin.
IA Conversacional SSE + OpenRouter

Consulta al Chatbot de Recomendaciones

  1. 1 Cliente abre chatbot en portal público: "¿Qué buffet recomiendan para 80 personas?"
  2. 2 Frontend abre conexión SSE a GET /api/ai/chat en NestJS.
  3. 3 Backend recupera contexto relevante vía pgvector (embeddings de carta y FAQs).
  4. 4 NestJS envía prompt enriquecido a OpenRouter (Laguna XS 2.1 o Llama 3 8B).
  5. 5 Respuesta se transmite token a token vía SSE — el usuario ve texto en tiempo real sin esperar respuesta completa.
